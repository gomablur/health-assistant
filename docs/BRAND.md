# ブランド・販売方針

2026-07-12 に議論して決めたブランディングとセールスの方針。機能開発の判断に迷ったら
ここに立ち返る。**決定事項は変えない前提で書くが、市場の反応で見直してよい。**

## ポジショニング(決定)

> **計測はもう終わっている(体重計とWatchがやってくれる)。足りないのは「意味」だ。**

- このアプリは**記録アプリではなく「解釈アプリ」**。記録・生データの保持はOSヘルス基盤
  (ヘルスケア / ヘルスコネクト)に任せ、うちは解釈に徹する
- **主軸は「体重を軸にした健康管理・伴走」**(2026-07-13決定)。体重が主役で、
  歩数・睡眠・心拍は体重の動きを説明する脇役。この順序を崩さない
- 中心の比喩は**「天気と気候」**。日々の体重の上下=天気、トレンド=気候。
  「今日は雨でも、季節は変わっていない」と毎朝ひとこと伝える存在
  (実装上も EWMAトレンド vs 実測ノイズの対比 = `src/coach/briefing.ts` の weightNoise ルール)
- もう一つの柱は**プライバシー**。解析は端末内・アカウント不要・サーバーなし・
  AIに渡るのは統計値のみ。大手の広告/データビジネスには構造的に真似できない

## 名前(決定)

- **日本語名: 「けさのからだ」**(App Store 検索で被りなしを確認済み、2026-07)
- 英語名は**保留**。当初案 "Karada" は健康カテゴリで既存アプリ多数
  (KARADAKarada、Karada Coaches、Karada 健康手帳、KARADA Indonesia)に加え、
  MTIの「CARADA」ブランドと音が同じで商標リスクがあるため**単体では使わない**。
  多言語対応(TODO.md)に着手するときに再検討。現候補: ローマ字のまま
  **"Kesa no Karada"**(被りなし・ブランド一貫)。"Kesa" 単体も柔術トラッカー等と被りあり
- **bundleIdentifier / package はストア公開後は変更不可**。改名作業はストア公開前が
  唯一のタイミング(app.json の name / slug / scheme / ios.bundleIdentifier /
  android.package を一括で変える)

## アイコン・ブランドカラー(決定)

- **アイコン**: 朝焼けのグラデーション背景+白い滑らかな一本の曲線(EWMAトレンド線)。
  ヘルス系の棚(赤ハート・緑十字・ティール)の中で浮く配色を狙う
- **ブランドカラー**: 朝の空の暖色(コーラル〜アンバー、基準候補 `#e8724a` 系)
- **操作系はOSネイティブ部品のまま、色だけブランドに揃える**(2026-07-12決定)。
  iOSのシステム青も、Androidの壁紙由来Material Youダイナミックカラーも、どちらも
  ブランドのコーラルで上書きする。形状・波紋・タッチ感はOS標準に任せる
- アプリ内テーマ(`src/constants/theme.ts`)もブランドに統一(2026-07-12):
  体重=ブランドコーラル `#e56638`、tint はライトのみ白文字コントラスト確保のため
  一段濃い `#c14e28`。心拍=ローズ・カロリー=アンバーへ移動(コーラルとの衝突回避)。
  変更時は dataviz スキルのバリデータで両モード再検証すること
- 生成は `scripts/gen-icons.mjs` のパイプラインを流用

## データが欠けているユーザーへの方針(2026-07-13決定)

全員が全メトリクスを埋めているとは限らない。原則は2つ:

- **聞かない。見て判断する。** 「体重計を持っていますか?」とは尋ねない(入力ゼロの原則に
  反するし、機材は後から増える)。直近30日にデータが1件でもあるかで機材の有無を推定する
- **体重がなければ成立しない、と正直に言う。** 体重が主役なので、トラッカーだけの
  ユーザーには空のチャートを見せず、`NeedsWeight` カードで「体重計と連携すると
  本領を発揮する」と伝える。責めない・急かさない、価値を説明するだけ

具体的な出し分け:

| 状況 | 挙動 |
|---|---|
| 体重なし(トラッカーのみ) | ヒーローと体重チャートを `NeedsWeight` に差し替え。歩数・睡眠のブリーフは出す |
| トラッカーなし(体重計のみ) | そのまま成立(歩数はスマホ単体で取れる)。睡眠・心拍のタイルは**隠す** |
| 一晩だけ計測が飛んだ | タイルは残して「未計測」と表示(機材がないのではなく、その日がないだけ) |
| 体脂肪率なし | 体組成の所見(`body-recomposition`)を出さないだけ。スマート体重計は必須ではない |

将来の拡張(BRAND方針の範囲内): 体重計を持たない層に降りていくなら「体重計なしモード」を
検討する。ただし主軸(体重ベースの伴走)をぼかしてまで対象を広げない。

## やらないことリスト(ブランドの一部としてストア説明文・サイトに明記する)

1. **手入力はさせない** — 計測はOSヘルスに任せる。入力ゼロが売り
2. **スコア・ランキング・通知爆撃をしない** — 不安を煽って開かせない
3. **生データをサーバーに送らない** — そもそもサーバーがない
4. **広告を入れない**

## ターゲットと成長戦略

現ターゲットは「スマート体重計+アクティブトラッカーを持つギークな健康オタク」。
層は薄いが、**個人開発は薄い層で十分成立する**(数百人の熱心なユーザー > 数万人の
無関心ユーザー)。しかもこの層は Apple Watch の普及とともに拡大し続けている。
層を広げたくなったら「手入力なし」原則を守ったまま、**体重計なしモード**
(iPhone単体でも入る歩数・睡眠だけでブリーフを出す)がライト層への入口になる。

打ち手(費用対効果順):

1. **Zenn記事** — 「毎朝の体重±1kgはなぜ気にしなくていいのか — EWMAで作る
   『トレンド体重』」のような技術記事がそのままマーケになる。記事末尾から
   **モックで全機能が動くWebデモ**(インストール不要の営業資産)へ誘導
2. **ASO** — ストア名+サブタイトル例:「けさのからだ — 体重のトレンド分析」。
   「体重 トレンド ヘルスケア連携 Apple Watch」を自然に含める
3. **レビュー導線** — 価値を感じた瞬間(例: ブリーフを14日連続で見た)にだけ
   `StoreReview.requestReview()` を出す
4. **Appleのフィーチャー(宝くじ)** — OSネイティブUI(NativeTabs・@expo/ui)+
   プライバシー設計は Apple が好んで紹介する型。確率は低いが期待値はゼロでない
- 有料広告はやらない(個人開発ではCACが合わない)。X運用は無理にしない
  (Zennだけで足りる)

## 収益化(決定)

**フリーミアム+買い切りPro(¥1,000前後)**。サブスク・広告はやらない。

- 根拠: サーバーコストがゼロ(LLMもユーザー自身のキー)なので、毎月課金する理由が
  構造的にない。**「サーバーを持たないので、あなたから毎月お金を取る理由がありません」**
  はサブスク疲れ市場への最強のコピーで、プライバシー方針とも一貫する
- 無料枠: ダッシュボード+基本ブリーフ。ブリーフは8日分以上のデータで真価が出るので、
  無料で2週間使ってから Pro を促す流れと相性が良い
- **Pro(買い切り)**: 全ブリーフルール・相関分析・AIコーチ・ウィジェット・エクスポート
- サブスクを再検討するのは、自前でLLMプロキシをホストする方針に転換した場合のみ

### 医療表現の線引き(有料化で重要度が上がる)

ブリーフの文言は「一般的なウェルネス情報」の枠を出ない。診断・治療を示唆しない
(最大でも「受診を勧める」止まり)。**新しいブリーフルールを追加するときのチェック項目**。

1. **改名+アイコン刷新**(小さく先に済ませて土台を固定する)
   - app.json の name / slug / scheme / bundleIdentifier / package を「けさのからだ」系へ
